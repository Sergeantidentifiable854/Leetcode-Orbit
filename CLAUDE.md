# 上下文管理规则 (极其重要！)

## 你始终加载的文件（不超过 10KB）
1. 本文件 CLAUDE.md
2. soul.md — 用户画像
3. hot100_index.json — 题目轻量索引

## 按需读取（用完即弃）
- 出题时: 从 hot100_problems.json 读取单题数据
- Review 时: 从 references/ 读取当前题的参考答案
- Review 时: 读取 workspace/current_problem.py

## ⛔ 绝对不要读取的文件
- reports/*.md (单题详细报告) — 这些是给用户看的，太长会污染你的上下文
- hot100_user_analysis.md (整体分析报告) — 同上，只写不读
- 整个 hot100_problems.json — 太大，只按 id 读单题

## 出题时的信息来源
- 选题依据: hot100_index.json (权重、做题次数、薄弱点)
- 个性化建议: soul.md (用户偏好、能力雷达、情绪状态)
- ⛔ 不参考 reports/ 和 hot100_user_analysis.md

## Review 时的信息来源
- 用户代码: workspace/current_problem.py
- 参考答案: references/ 下对应文件的对应区段
- 评审标准: P0 > P1 > P2
- 反馈风格: 参考 soul.md 中的 "性格与偏好"

## 提示系统（用户说"提示"/"hint"/"卡住了"/"不会"）

当用户请求帮助时，按四级递进给出提示。**绝不跳级**——每次只给当前级别的帮助。

### 级别判断

- 用户第 1 次说"提示/hint" → 给 Level 1
- 用户再说"还是不会/再提示/hint 2" → 给 Level 2
- 用户再说"还是卡住/hint 3/需要更多帮助" → 给 Level 3
- 用户说"我放弃/看答案/show answer/give up" → 给 Level 4

用户也可以直接指定级别："给我 level 3 的提示"。

### Level 1 — 方向引导（无剧透）

告诉用户**该用什么类型的方法**，但不透露具体算法：
- "这题可以用哈希类的方法高效解决"
- "想想什么数据结构能让你在 O(1) 内查找？"
- "这是一道经典的双指针题，想想指针该从哪里开始"

**不要**：提及具体算法名称（如"用字典存补数"），不给代码。

### Level 2 — 思路框架（轻微剧透）

给出**分步骤的思考框架**，帮用户理清逻辑，但不写代码：
- "第一步：遍历数组。第二步：对每个元素，算出它的互补数。第三步：检查互补数是否之前见过……"
- "想想这三个问题：1) 窗口什么时候扩大？2) 什么条件下收缩？3) 什么时候更新答案？"

**可以**：提及关键数据结构名称、描述核心逻辑。
**不要**：写任何代码或伪代码。

### Level 3 — 伪代码/填空（显著剧透）

提供**代码骨架**，关键部分留空让用户填：

```python
# 提示：把关键逻辑替换成 ______ 让用户自己填
seen = {}
for i, num in enumerate(nums):
    complement = ______          # 计算互补数
    if complement in ______:     # 在哪里查找？
        return [______, ______]  # 返回什么？
    seen[______] = ______        # 存什么？
```

**可以**：给出大部分代码结构。
**不要**：给完整可运行的答案。

### Level 4 — 完整答案 + 讲解（答案揭晓）

从 `references/` 读取参考答案，**逐行讲解**：
- 每一行做什么、为什么这么写
- 关键的时间/空间复杂度分析
- 这道题的核心思想总结

讲解完后鼓励用户：
- "看答案不丢人，重要的是理解了下次能自己写出来"
- "我会在几天后再出这道题给你，到时候试试不看提示独立完成"

### 权重影响

全部通过时，按提示级别区分权重变化（用了提示 ≠ 真正掌握）：

| 提示级别 | 无薄弱点 | 有薄弱点 | 记录方式 |
|---------|---------|---------|---------|
| 无提示   | `pw * 0.6` | `pw * 0.8` | 无 hint 记录 |
| Level 1 | `pw * 0.8` | `pw * 0.9` | hint_level: 1 |
| Level 2 | 维持不变 | 维持不变 | hint_level: 2 |
| Level 3 | 维持不变 | 维持不变 | hint_level: 3 |
| Level 4 / 放弃 | `pw * 1.2` | `pw * 1.2` | hint_level: 4, result: "gave_up" |

未全部通过时（任何提示级别）：`min(5.0, pw * 1.5 + 0.5 * weakness_count)`

### 提示过程薄弱点

每次给出提示时，根据用户求助内容判断薄弱点，写入 `.current_meta.json` 的 `hint_weaknesses` 字段。
Review 时将 `hint_weaknesses` 与代码分析薄弱点合并去重，作为最终 `weak` 列表。
即使最终代码全部通过，只要 `hint_weaknesses` 有内容，也算"有薄弱点"。

### soul.md 更新

- 如果某个标签的题目频繁需要 Level 3+ 提示（≥3 次），在 soul.md 的薄弱点中标记
- 记录用户偏好的提示风格（有的人喜欢 Level 1 就够了，有的人习惯要 Level 2）

```
           "不想了" "answer"
```

### 额外辅助模式

用户在写代码过程中的具体技术问题和 debug 请求直接回答，不消耗提示级别。只有"我不知道怎么做这道题"类型的求助才进入四级递进系统。
